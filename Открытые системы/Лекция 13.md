# Об открытых системах. OSE (Open System Enviroment)

* **OSE** -- модель, описывающая функциональную среду открытых систем.
* Ее также называют Reference Model (да, она была эталонной)
	* Эталонная модель -- это структурированное множество понятий и взаимосвязей для некоторой предметной области. Если проще, то это просто концептуальное структурированное обобщенное описание.  
* Разработана была группой 1003.0 POSIX в рамках IEEE (комитет по стандартам)
* OSE вводит общие соглашения и понятия, которые позволяют применять разработанное для одной системы
* В описании модели используются два типа элементов:
	* Логические объекты:
		* Application -- логические объекты по типу ППО (прикладного ПО) 
		* Platform -- среда выполнения ПО
		* Enviroment -- внешнее окружение
	* Интерфейс -- взаимосвязь между системой и окружением
	* В общем, это должно выглядеть так:

* Суть общая в том, что между программой и внешней средой напрямую быть не должно. Только через платформу.
* Приложения так же могут взаимодействовать между собой, но так же через платформу.
* Приложение можно разбить на следующие части:
	* Program - программные компоненты, сам исполняемый код. Его принято делить на высокоуровневую и низкоуровневую часть. 
	* Данные - переменные и прочие обрабатываемые данные.
	* Документация - встроенный документ. Например, описание меню, пункт Help итд.
* В состав External Enviroment входят:
	* Люди, которые как-то получают от платформы данные с ними как-то работает.
	* Информационное обеспечение
	* Коммуникационное обеспечение - используется для коммуникации между другими системами.

* API так же разбивается на несколько сегментов:
	* System Services. Используется для работы со службой операционной системы. Примером можно назвать порождение одним процессом другого. 
	* Communication Service. Сервис для коммуникации между приложениями.
	* Information Service. 
	* Human/Computer Service.
		* Стоит отметить, что все пункты, кроме первого, так же есть и в интерфейсе взаимодействия с внешней средой.
		* Платформы могут быть разными, могут быть одинаковыми. Если же платформы застакать, то получится уже более комплексная сложная платформа - кластер.

* У платформы есть следующие компоненты (службы):
	* Вширь:
		* Службы операционной системы.
		* Службы человекомашинного интерфейса
		* Службы обмена данными
		* Службы машинной графики
		* Службы сетевого обеспечения (для взаимодействия с другими платформы)
	* Вглубь:
		* Службы поддержки разработки ПО. Позволяет, разрабатывая программу для одного устройства, компилировать двоичные коды для других устройств.
		* Службы защиты информации. Методы шифрования, идентификации и прочие вопросы безопасности
		* Интернационализация. Позволяет решать одни бизнес-задачи (НАПРИМЕР) представителями разных стран и фирм.
		* Службы поддержки распределенной системы. Используется для управления кластером: какую задачу кому направить.

# OSI (Open System Interconnection)
* Появилась раньше чем OSE, где-то в 1982 году.
* Был введен ISO (International Standart Organization)
	* Формально, сетями начали заниматься еще в 60-х.
	* А стали ими заниматься, чтоб от МБР и БРСД защищаться (надо же как-то обмениваться инфой между разными точками наблюдения, командным центром и пусковыми точками). как обычно, все началось с военной отрасли.
	* И эта вся штука переросла в интернет
* Введено было для стандартизации взаимодействия между различными открытыми системами.
* Тогда открытую систему понимали как расширяемую систему, которую не недо было кардинально переписывать для этого самого расширения.
* По итогу эта модель получилась 7-уровневой. Хотя фактически у нее 8 уровней. А случилось это из-за недоработки.
* В ее практической реализации были серьезные проблемы. Так что была разработана еще одна сетевая модель - стек протоколов TCP/IP (4-уровневая модель, но в ней только 3 уровня специфицированы. 4 уровень - свободное плавание).
* Ну а чтоб разработка OSI не была сделана напрасно, именно ее используют для изучения в учебных заведениях. В TCP/IP их используют так же, просто их там, считай, перегруппировали.

* Работа осуществляется по схеме [[Операционные системы/Лекция 9#Регламент обмена данных в сети|клиент-сервер]].
* Есть еще и 3-уровневая версия, где есть прослойка между клиентом и сервером - будет сервер, который обратится к другому серверу как клиент.
* В этих моделях используется как раз 3-уровневая. Решили, что обращения клиента серверу должны быть короткими (одна команда, да и та, из API). Если же запрос от клиента серверу длинный и сложный, то трехуровневая система будет применяться многократно.
* Отношения могут быть вертикальными (клиент обращается к своему серверу, который обращается к другому серверу итд) и горизонтальными (они же являются протоколами).
