# Пакеты (продолжение)

`synaptic` - это графический менеджер пакетов для `apt`.
* Скачивание пакетов осуществляется с репозиториев. Эти самые репозитории представляют собой БД.
* Такие БД могут быть как сетевыми, так и на отдельных носителей.
* Обычно, под свою версию дистрибутива есть свой глобальный сетевой репозиторий
* Есть еще и PPA-репозитории - репозитории, которые создаются авторами проектов, которые разрабатываются обособлено от дистрибутива  (и от глобального репозитория также).
* `synaptic` 

Для управления ПО используются следующие компоненты:
	* программные пакеты (бинарники и исходники)
	* Менеджеры пакетов (dpkg, apt-команды, apt, `aptitude` (графический менеджер с псевдографикой в консоли), synaptic)
		* Администратор может разрешать/запрещать установку тех или иных программ. Например, в Astra Linux для этого используется `Kiosk` (с помощью него вообще можно управлять доступом ко всем программам)
		* В менеджере пакетов можно выполнить команду `upgrade`, которая обновит все имеющиеся пакеты (при наличии обновления), или `full-upgrade`, которая может обновить вообще все компоненты (да, даже ядро).
			* `update` позволяет синхронизировать списки доступных пакетов из репозиториев.
			* `upgrade` - если среди установленных пакетов вышли обновленные версии, то они будут обновлены
	* Репозитории
	* БД

Про программные пакеты:
	* Правильным считается следующий нейминг: `имя_пакета-версия-архитектура.deb`
	* Сам по себе пакет представляет собой архив файлов (не обязательно сжатый)
	* Такие архивы в виде исходников
	* Структура бинарного пакета:
		* данные (файлы) пакета
		* метаданные
	* Метаданные хранятся данные о связях с другими пакетами:
		* Depends - базовый минимум. Без них пакет не сможет нормально функционировать
		* Recommends - среднеклассовое. Пакет сможет без них
		* Suggests
	* Есть еще метапакеты (пустышки, которые отправляются в Depends, в его метаданные хранятся названия всех необходимых пакетов (они так же лежат в Depends)) и виртуальные пакеты (фактически, не пакет, содержит название группы схожих по функциональности пакетов)
		* Метапакеты могут использоваться, например, для удобной установки всех Depends, разбивая все пакеты необходимые пакеты, НАПРИМЕР, по группам.

Про репозиторий:
	* Репозиторий - централизованное хранилище пакетов
	* Репозиторий содержит данные (deb-файлы, в каталоге pool/) и метаданные (описание репозитория и пакетов, в каталоге dist/)
	* По типу лицензии репозиторий делится на компоненты:
		* main - все основные компоненты
		* non-free - проприетарное ПО
		* non-free-firmware - частный случай проприетарного ПО (берет)

# Графика

* Первые версии графики работали в полноэкранном режиме
	* Изображение генерировалось и выводилось растровым

* Затем, где-то к 1984 году три студента придумали X Window, где и впервые было введено понятие окна. Оно позволило запускать и выводить несколько приложений на одном экране (каждое приложение выводилось в своем "виртуальном образе" - окне)
	* Эту штуку разработали под UNIX
	* X Window - это протокол взаимодействия клиента и сервера
		* Сервер - это тот программный модуль, который отвечал за отрисовку изображения на экране.
			* Вводятся понятия кнопок, привязанных к окну (регулировать это самое окно), программы по отрисовке графическому 
			* К серверу еще относятся обработки указаний от устройств ввода
		* Клиент - это различные приложения.
			* Клиенты управляют отрисовкой. Сервер просто получает команду на отрисовку с такими-то параметрами. Но эту команду с параметрами отдает клиент.
			* Клиентами являются приложения, менеджер файлов и панель управления
			* Клиентов принято называть X Client
			* Протокол взаимодействия между клиентом и сервером является сетевым.
				* Например, это позволяет нам выполнять команду на другом устройстве, при этом просматривая ее выполнение и отрисовку у себя.
				* Локально (в рамках одного устройства) это все так же работает
	* Следующим стал X Free 86 - бесплатная реализация X Window для UNIX. Ныне используется X11
	* Есть реализация X Window и на Windows. MIcrosoft ее не поддерживает, этим занимаются другие.
	* Архитектуры процессоров не имеют к этому никакого отношения.
	* Нынче отрисовкой занимаются различные оконные менеджеры. Они работают как клиенты для X Window
	* Главная библиотека для работы графики - xlib.
	* Для написания программ используется OpenGL (по крайней мере, принят в линуксе именно он).
		* В Microsoft пошли по другому пути - они разработали свой DIrectX
